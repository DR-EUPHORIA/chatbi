# ChatBI Mini — 项目设计书

> 基于 LangChain/LangGraph 的对话式智能数据分析平台（课设版）
> 参考：阿里巴巴内部 FBI 平台 ChatBI 产品

---

## 一、项目概述

### 1.1 项目目标

构建一个 **Mini 版 ChatBI**，让用户通过自然语言对话完成"从提问到拿到一份完整数据分析报告"的全流程。

**一句话定义**：ChatBI Mini = LangGraph 编排的多步骤 Agent + 精调 Prompt + 数据分析工具链 + 好看的前端。

### 1.2 项目定位

- **课设项目**，不追求企业级特性（权限、多租户、审计等）
- **完全参考阿里 ChatBI**，做一个功能完整的 Mini 版
- **高标准严要求**，核心功能一个不少

### 1.3 不做什么

| 不做 | 原因 |
|------|------|
| 企业级权限管理、多租户 | 课设不需要 |
| 协作分享、评论标注 | 非核心 |
| ODPS / MaxCompute 支持 | 环境不便 |
| 安全沙箱 | 暂不考虑 |
| MCP 协议 | 除非 LangChain 有现成支持 |
| 定时报告 / 订阅推送 | 后期再说 |

## 二、核心设计原则

### 2.1 门控优先，拒绝浪费

Agent 每一步都消耗 Token。如果用户指令模糊就直接跑，大概率结果不对还浪费资源。

**设计**：在 Agent 入口设置 **意图门控 Node**，对用户输入做"可执行性评估"：
- 指令太模糊 → 打回，给出引导性追问
- 指令和数据集不匹配 → 打回，告知用户
- 指令清晰可执行 → 放行，进入执行链路

### 2.2 执行过程必须透明

每一个 Node 的执行状态、输入、输出、耗时，都要实时推送到前端。用户能看到：
- "正在理解你的问题..."
- "已匹配到 3 张相关表"
- "生成 SQL：`SELECT ...`"
- "SQL 执行成功，返回 1,234 行数据"
- "正在生成图表..."
- "报告已生成"

**技术实现**：后端通过 SSE（Server-Sent Events）流式推送。

### 2.3 规划过程细粒度

Agent 在执行前，先输出一个 **执行计划（Plan）**：
- 计划粒度细到每一步做什么、用什么数据、预期输出什么
- 用户可以确认、修改、跳过某步

### 2.4 全链路 I/O

**输入侧**（什么都能读）：
- 📊 Excel / CSV 文件上传
- 📄 PDF 文件上传（提取表格和文本）
- 🗄️ 数据库直连（MySQL、PostgreSQL、SQLite）

**输出侧**（什么都能写）：
- 🌐 HTML 报告（带图表、带样式、可分享）
- 📝 Markdown 报告
- 📊 Excel 文件（带格式、多 Sheet）
- 📊 ECharts 交互式图表
- 🖥️ 数字化大屏页面
- 📑 PPT 演示文稿（集成 AIPPT）

### 2.5 撑门面的 Demo

首页不能是空白聊天框，必须有预置的高质量 Demo：
- Demo 让用户一进来就知道"这个东西能干什么"
- Demo 同时也是 Prompt 引导——用户模仿着写出更好的 Prompt
- 数字化大屏 Demo 是最大亮点

### 2.6 充分参考社区开源项目

- Text-to-SQL：参考 Vanna.AI、WrenAI
- 数据分析 Agent：参考 JoyAgent、Open Interpreter
- 大屏可视化：参考 DataV 组件库、AJ-Report、BigDataView
- 报告生成：参考 Datapane
- AIPPT：参考 python-pptx

### 3.2 Agent 工作流（LangGraph DAG）

```
[用户输入]
    │
    ▼
┌─────────────────┐
│  意图门控 Node   │──→ 不清晰？→ [引导追问 Node] → 等待用户补充 → 回到入口
│  (Gate)          │
└────────┬────────┘
         │ 清晰
         ▼
┌─────────────────┐
│  意图路由 Node   │──→ 闲聊？→ [直接回复]
│  (Router)        │──→ 文件分析？→ [文件分析子流程]
└────────┬────────┘──→ 问数分析？→ 继续 ↓
         │
         ▼
┌─────────────────┐
│  规划 Node       │──→ 输出执行计划，等待用户确认
│  (Planner)       │
└────────┬────────┘
         │ 用户确认
         ▼
┌─────────────────┐
│  Schema 检索     │──→ 从元数据中找到相关表和字段
│  Node            │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  歧义检测 Node   │──→ 有歧义？→ [反问用户 Node] → 等待选择 → 回到上一步
│  (Clarifier)     │
└────────┬────────┘
         │ 无歧义
         ▼
┌─────────────────┐
│  SQL 生成 Node   │──→ 基于 Schema + 语义层 + Few-shot 生成 SQL
│  (SQL Generator) │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  SQL 执行 Node   │──→ 报错？→ [SQL 修正 Node] → 重试（最多3次）
│  (SQL Executor)  │
└────────┬────────┘
         │ 成功
         ▼
┌─────────────────┐
│  数据分析 Node   │──→ 用 Python/Pandas 做进一步计算
│  (Analyzer)      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  可视化 Node     │──→ 根据数据特征选图表类型，生成 ECharts 配置
│  (Visualizer)    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  报告组装 Node   │──→ 组装 HTML / MD / Excel / PPT
│  (Reporter)      │
└────────┬────────┘
         │
         ▼
[输出给用户：图表 + 报告 + 数据文件]
```

## 四、功能模块详细设计

### 4.1 模块一：Agent 引擎

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 多轮对话 | 支持上下文记忆，追问、补充、修正 | P0 |
| 意图门控 | 指令不清晰时打回 + 引导追问 | P0 |
| 意图路由 | 识别意图（问数、报告、文件分析、闲聊），路由到对应子流程 | P0 |
| 歧义澄清 | 问题模糊时主动反问用户（Human-in-the-loop） | P0 |
| 执行过程可视化 | 前端展示每一步 Plan/Action/Observe | P0 |
| 细粒度规划 | 执行前输出计划，用户确认后执行 | P1 |
| 模型可切换 | 支持配置不同 LLM（Qwen、Claude、GPT 等） | P1 |
| 增量对话 | 在上次结果基础上增量修改，不从头跑 | P1 |

### 4.2 模块二：数据接入层

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 数据库连接 | 支持 MySQL、PostgreSQL、SQLite | P0 |
| 文件上传（Excel/CSV） | 上传后自动解析为 DataFrame | P0 |
| PDF 文件解析 | 提取表格和文本内容 | P1 |
| 元数据自动采集 | 连接数据源后自动采集表结构、字段类型、注释 | P0 |
| 数据集管理 | 创建、管理多个数据集，圈定分析范围 | P1 |
| 多数据源 Join | 跨数据源关联查询（如 Excel + 数据库） | P1 |

### 4.3 模块三：智能问数（Text-to-SQL）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 自然语言转 SQL | 核心能力，基于 Schema + 语义生成 SQL | P0 |
| 语义层 / 指标字典 | "业务术语 → 物理字段"映射表 | P0 |
| SQL 自动修正 | 执行报错时自动修正，最多重试 3 次 | P0 |
| Few-shot 示例库 | "问题 → SQL"示例对，RAG 检索提升准确率 | P1 |
| 业务术语词条 | 配置术语解释，注入 Prompt | P1 |
| 反馈学习 | 用户点赞/踩，好 case 沉淀到示例库 | P2 |

### 4.4 模块四：可视化与图表

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 智能图表推荐 | 根据数据特征自动选择图表类型 | P0 |
| ECharts 渲染 | 前端使用 ECharts 渲染，支持交互 | P0 |
| 常规图表 | 柱状图、折线图、饼图、散点图、环形图 | P0 |
| 创意图表 | 地图热力图、水波图、仪表盘、桑基图、漏斗图、雷达图、词云、K线图 | P1 |
| 数字化大屏 | 内置 3-5 套大屏模板，科技感 UI | P0 |
| 图表配色方案 | 内置多套专业配色 | P1 |

### 4.5 模块五：报告与导出

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 自动生成分析报告 | 标题 + 摘要 + 图表 + 结论的完整报告 | P0 |
| HTML 报告导出 | 生成可分享的 HTML 页面 | P0 |
| Markdown 导出 | 导出 MD 格式文档 | P0 |
| Excel 数据导出 | 查询结果和明细数据导出为 Excel | P0 |
| PPT 生成 | 集成 AIPPT，自动生成演示文稿 | P1 |
| 报告分享链接 | 生成链接，他人可查看 | P2 |

### 4.6 模块六：知识库与调优

| 功能 | 说明 | 优先级 |
|------|------|--------|

## 五、技术栈

| 层次 | 技术选型 | 说明 |
|------|----------|------|
| **Agent 框架** | LangChain + LangGraph | 多步骤编排、条件分支、Human-in-the-loop |
| **LLM** | 通过 LangChain 接入 | 默认 GPT-4o / Qwen，可切换 |
| **后端** | Python FastAPI | 高性能异步框架，原生支持 SSE |
| **前端** | React + TypeScript + Ant Design | 阿里系设计语言，和 ChatBI 风格一致 |
| **图表** | ECharts | 最强大的开源图表库 |
| **大屏组件** | DataV React 组件库 | 科技感边框、装饰、飞线图 |
| **数据库连接** | SQLAlchemy | 统一多数据库方言 |
| **文件处理** | Pandas + openpyxl + pdfplumber | Excel/CSV/PDF 读写 |
| **PPT 生成** | python-pptx | PPT 文件生成 |
| **向量库（RAG）** | FAISS 或 Chroma | 存储 Few-shot 示例 |
| **通信协议** | SSE (Server-Sent Events) | 实时推送执行过程 |

## 八、Prompt 设计要点

### 8.1 总体原则

- 每个 Node 的 Prompt 都要**精心打磨**，这是整个系统的灵魂
- Prompt 中必须包含：角色设定、上下文信息、输出格式约束、边界条件
- 使用 Few-shot 示例提升准确率
- 严格约束输出格式（JSON Schema），方便前端解析

### 8.2 关键 Prompt 设计

#### 门控 Prompt（Gate）
```
你是一个数据分析需求评估专家。
评估用户的输入是否足够清晰，能否直接进行数据分析。

评估维度：
1. 是否明确了要分析的指标/数据？
2. 是否明确了时间范围？
3. 是否明确了分析维度？

如果不够清晰，输出需要用户补充的信息。
如果足够清晰，输出 PASS。
```

#### SQL 生成 Prompt
```
你是一个专业的数据分析师。根据以下信息生成 SQL：
- 数据库方言：{dialect}
- 相关表结构：{schema}
- 业务术语：{glossary}
- 相似问题的历史 SQL（Few-shot）：{examples}
- 用户问题：{question}

要求：
1. 只使用提供的表和字段，不要编造
2. 注意时间字段的格式
3. 输出标准 SQL，不要使用特定方言的语法
```

#### 可视化 Prompt
```
你是一个数据可视化专家。根据查询结果数据，推荐最合适的图表类型并生成 ECharts 配置。

数据特征：{data_summary}
用户偏好：{user_preference}

图表选择规则：
- 时间序列 → 折线图
- 分类对比 → 柱状图
- 占比分析 → 饼图/环形图
- 多维度对比 → 雷达图
- 地理分布 → 地图
- 流程转化 → 漏斗图

输出格式：标准 ECharts option JSON
```

## 十、风险与应对

| 风险 | 应对 |
|------|------|
| LLM 生成的 SQL 准确率不够 | 语义层 + Few-shot + 自动修正三重保障 |
| Token 消耗过大 | 门控机制 + 缓存 + 精简 Prompt |
| 前端工程量大 | 充分利用 Ant Design + DataV 组件库 |
| 大屏模板开发耗时 | 直接参考开源模板，改造适配 |
| PPT 生成质量不高 | 使用 python-pptx + 预置模板，降低自由度 |
